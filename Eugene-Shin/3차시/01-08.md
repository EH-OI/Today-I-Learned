## 3차시(1/9) - Spring vs Spring Boot

### 1. Spring Boot란 무엇인가
- Spring 기반 프레임워크 쓰기 쉽게 해 줌
- 운영에 필요한 기능
	- starter : 내가 어떤 것 implementation하고 있는지 보고 의존성 있는 것 자동으로 싹 가져와서 붙여줌
	- auto configuration : yml 바탕으로 자동으로 xml config 만들어 등록해 줌
	- 자동 로깅, 런타임 자동 실행(JAR) 등

### 2. Spring Framework와 Spring Boot의 차이
- Spring
	- Component Scan 다 붙여야, basePackage(componentScan 시작점) 수동 등록 등 필요
	- Bean 이름 다 수동으로 등록해 줘야 함
	- properties 파일
		- 기본적으로 value 값을 다 String 취급해서 잘못 변환할 수도 있음
		- key 값이 중복되어도 체크 못하고 맨 마지막 값을 씀
	- 로깅하고 싶으면 log4j로 등록해야 함(일정 버전 이상으로는 보안 위험 있어 Slf4j 대신 사용)
- Spring Boot
	- 대체로 자동으로 해 줌
	- yml을 내장 팩토리 통해 java config로 만들어 줌
		- 들여쓰기에 민감(틀리면 표시해 줌)
		-  auto configuration
			- application으로 시작하는 yml과 properties 파일을 찾아 java config 파일을 자동으로 만듦
			- application-local.yml, application-deploy.yml 등 여러 개 만들어서 매핑으로 원하는 config 만들어 실행하도록 할 수 있음
		- 의존성 있는데 빠져 있거나 부족한 부분 있으면 자동으로 등록해 주기도 함
			- `jpa:hibernate:ddl-auto:update`인데 db에 아무것도 없는 경우 hibernate가 자동으로 create문 쏴 줌

### 3. JAR vs WAR
 - Spring Boot의 JAR(기본 JAR와 좀 다른 개념)
	 - 이것만 있으면 끝
	 - JAR 파일 하나가 서버 하나
	 - 실행 가능한 application(서버 하나), 기본적으로 Tomcat 내장되어 있으며 치환 가능
	 - Tomcat의 Servlet Container 사용해 Dispatcher Servlet 자동 관리
	 - JVM이 실행시킴, 이 때 Tomcat도 초기화됨
	 - 서버 하나 추가될 때마다 Tomcat 하나 추가해야 함
	   (EC2 인스턴스에 올릴 경우 리소스 더 요구하긴 함)
	 - 각 서버가 독립적이라 하나 죽어도 영향 미치지 않음
 - WAR
	 - Spring
	 - 혼자 실행이 안 됨, 독립적인 서버가 될 수 없음
	 - JVM 없고 자기 스스로 실행시키지 못함, 그래서 main 파일이 없음
	 - web.xml이나 java 기반 config 만들어야 Tomcat이 그거 보고 초기화할 수 있음
	 - 외부 Tomcat에 의존적
	 - Tomcat 하나로 여러 서버 한 번에 올릴 수 있음(다만 이러면 버전 관리 각기 따로 못 함)

### 4. Gradle vs Maven
- 빌드 툴을 작성하기 위한 DSL(도메인 특화 랭귀지)
- Maven
	- 정해진 규칙을 무조건 따라야 함
	- build lifecycle이 정해져 있음
	- 예전부터 써오던 거라 관련 레퍼런스가 많음
	- xml이라 보기 불편
	- 좀 느림
- Gradle(Groovy)
	- build lifecycle이 정해져 있음
	- Java config랑 똑같음(Java로 만들어 줌), 이 자체로도 실제 실행 가능한 Java 코드
	- 유연하게 커스터마이즈 가능
	- 쓰기 편하고 성능이 Maven보다 좋음
